# web-clipper 新機能提案 - 2026-04-01

## エグゼクティブサマリー

2026年のWebクリッパー市場ではAI統合・ブラウザ拡張・ハイライト機能が標準化しつつある。本プロジェクトはClean Architecture/DDDに基づく堅牢なバックエンドを持つが、ブラウザ拡張の不在とAI機能の限定性が競合との最大のギャップである。本提案では、ブラウザ拡張の開発、AI自動タグ付け、ハイライト・アノテーション機能を軸に8つの機能を提案する。

## 現状の機能セット

- **記事クリッピング**: URLからメタデータ（OGP、タイトル、説明文）を自動抽出し保存
- **AI要約生成**: Cloudflare AI（llama-3.1-8b-instruct）による記事要約
- **タグ管理**: カスタムタグによる記事分類（多対多リレーション）
- **検索・フィルタ**: タイトル/説明文の全文検索、ソース/タグ/既読状態フィルタ、カーソルページネーション
- **ソース自動分類**: 11種のソース（Twitter, Qiita, Zenn, Hatena, GitHub, ClassMethod, Medium, Note, Dev.to, StackOverflow, Other）
- **認証**: ユーザー名/パスワード + GitHub OAuth、Cookie ベースセッション管理
- **PWA対応**: Web Share Target APIによるOS共有メニューからの記事追加
- **ダーク/ライトモード**: テーマ切り替え・永続化
- **View Transitions**: ページ遷移アニメーション
- **API ドキュメント**: OpenAPI仕様 + Scalar UIによるAPIリファレンス

### 2026年のトレンド

1. **AIアシスタント統合**: 単純な要約を超え、チャット型のドキュメント質問応答（Readwise Ghostreader）やコンテキスト理解に進化
2. **ハイライト・アノテーション**: テキスト選択 → ハイライト保存が標準機能化。再訪問時の可視化も重要
3. **ブラウザ拡張の必須化**: ワンクリック保存がUXの基本。全主要競合がChrome/Firefox/Safari拡張を提供
4. **プライバシー・ローカルファースト**: Obsidianのようなローカル保存・オープンフォーマット（Markdown）志向
5. **自動整理・スマートタグ**: AI による自動カテゴリ分類・タグ付けでユーザーの手間を削減
6. **マルチメディア対応**: YouTube動画、PDF、EPUB、ニュースレターなどURL以外のソース対応

## 新機能提案

### 優先度: High（すぐに着手すべき）

#### 提案 1: ブラウザ拡張（Chrome Extension）

- **概要**: Chrome拡張機能を開発し、閲覧中のページをワンクリックでクリッピング。選択テキストの右クリックメニューからの保存にも対応する。
- **ユーザー価値**: 現在はWeb UIから手動でURLを入力する必要があるが、拡張により閲覧フローを中断せずに記事を保存できる。全主要競合が提供している機能であり、UXの根幹を成す。
- **実装難易度**: Medium
- **技術的アプローチ**:
  - TypeScriptでChrome Extension Manifest V3を採用
  - `packages/extension` としてmonorepoに追加
  - 既存の `POST /api/articles` エンドポイントをそのまま利用
  - `@web-clipper/shared` のZodスキーマを拡張側でも再利用し型安全性を確保
  - Popup UIはVue 3で構築（既存スタックの統一）
  - Content Scriptで選択テキスト・ページメタデータを取得
  - 認証はCookieベースのセッションを拡張からも利用（same-origin）、またはAPIトークン方式を追加
- **参考**: [Obsidian Web Clipper (GitHub)](https://github.com/obsidianmd/obsidian-clipper), [Summer Web Clipper](https://github.com/merrickluo/summer-web-clipper)

#### 提案 2: AI自動タグ付け・スマート分類

- **概要**: 記事クリッピング時にAIが記事内容を解析し、既存タグから最適なタグを自動提案、または新規タグを推薦する機能。
- **ユーザー価値**: 手動タグ付けの手間を大幅に削減。記事が増えるほどタグ付けが面倒になる問題を解決し、整理された状態を維持できる。競合（Remio, Stacks, Notix）が実装済みの機能。
- **実装難易度**: Low
- **技術的アプローチ**:
  - 既存のCloudflare AI binding（`env.AI`）を活用
  - `clipArticle` コマンド実行時に、抽出済みコンテンツ（Readability）+ 既存タグリストをプロンプトに渡す
  - ドメイン層に `TagSuggester` ドメインサービスインターフェースを追加
  - インフラ層に `WorkersAiTagSuggester` を実装
  - APIレスポンスに `suggestedTags: string[]` を追加し、フロントエンドで確認UIを表示
  - `@web-clipper/shared` にスキーマを追加
- **参考**: [Remio AI Features](https://www.remio.ai/post/top-web-clipper-tools-compared-find-your-perfect-match-in-2026), [Stacks AI-powered recall](https://chromewebstore.google.com/detail/stacks-web-clipper-and-ai/nfjbjlpkfimhobegkcoekpkdlokjkcfj)

### 優先度: Medium（次のマイルストーンで検討）

#### 提案 3: ハイライト・アノテーション機能

- **概要**: 記事本文内の重要な箇所をハイライト（選択・色付け）し、メモ（アノテーション）を付与できる機能。現在の `memo` フィールド（記事全体に1つ）を拡張し、本文中の特定箇所に紐づくアノテーションを実現。
- **ユーザー価値**: Readwise Reader、Obsidian Web Clipper、Web Highlightsが全て提供する機能。記事を「読んで終わり」ではなく「知識として蓄積」するための中核機能。
- **実装難易度**: High
- **技術的アプローチ**:
  - D1に `highlights` テーブルを追加（article_id, text_content, start_offset, end_offset, color, note, created_at）
  - ドメイン層に `Highlight` エンティティ、`Article` 集約にハイライトを含める（ただし集約が大きくなりすぎる場合は別集約に分離）
  - フロントエンドで記事詳細ページにテキスト選択 → ハイライト保存UIを実装
  - 将来的にブラウザ拡張のContent Scriptと連携し、元ページ上でのハイライトも可能に
- **参考**: [Readwise Reader Highlights](https://readwise.io/read), [Web Highlights](https://web-highlights.com/blog/top-7-web-clipper-chrome-extensions-for-capturing-websites/)

#### 提案 4: 全文コンテンツ検索

- **概要**: 現在の検索はタイトル・説明文のみが対象だが、Readabilityで抽出した記事本文を保存し、全文検索を可能にする。
- **ユーザー価値**: 「あの記事に書いてあった○○」を見つけたいとき、タイトルだけでは不十分。記事本文をフルテキスト検索できることで知識ベースとしての価値が飛躍的に向上。
- **実装難易度**: Medium
- **技術的アプローチ**:
  - D1に `article_content` テーブルを追加（article_id, markdown_content, plain_text）
  - `clipArticle` コマンド内で既に `ContentExtractor` がMarkdownを生成しているので、それをDBに永続化
  - SQLiteのFTS5（Full-Text Search）を活用。D1がFTS5をサポートしていない場合はLIKE検索 or Cloudflare AI Embeddingsによるセマンティック検索を検討
  - 検索クエリAPIに `contentSearch` パラメータを追加
- **参考**: [Readwise Reader Full-text Search](https://readwise.io/read)

#### 提案 5: RSS フィード購読

- **概要**: RSSフィードURLを登録し、新着記事を自動でクリッピングする機能。Cron Triggerで定期実行。
- **ユーザー価値**: お気に入りのブログ・メディアの新着記事を手動保存する手間を省く。Readwise Readerが統合済みの機能。
- **実装難易度**: Medium
- **技術的アプローチ**:
  - D1に `feeds` テーブル（id, url, title, last_fetched_at）と `feed_entries` テーブルを追加
  - Cloudflare Workers の Cron Trigger（`wrangler.json` の `triggers.crons`）で定期実行
  - 既存の `clipArticle` コマンドを再利用してフィードエントリをクリッピング
  - ドメイン層に `Feed` 集約を新設
- **参考**: [Readwise Reader RSS](https://docs.readwise.io/reader/docs)

### 優先度: Low（長期的に検討）

#### 提案 6: AIチャットアシスタント（Ghostreader的機能）

- Readwise ReaderのGhostreaderのように、保存した記事に対して質問応答できるチャット機能
- Cloudflare AIのLLMを活用し、記事コンテンツをコンテキストとして渡す
- 「この記事の主要な論点は？」「この技術の利点と欠点は？」などの質問に回答

#### 提案 7: エクスポート機能（Obsidian / Notion連携）

- 保存記事をMarkdown形式でエクスポート（Obsidian Vault連携）
- Notion APIを通じたNotionデータベースへの同期
- 一括エクスポート（JSON / CSV / Markdown ZIP）

#### 提案 8: マルチメディア対応（YouTube・PDF）

- YouTube URLからの動画メタデータ + 自動文字起こし保存
- PDFファイルのアップロード・テキスト抽出・保存
- `Source` 値オブジェクトにyoutube, pdfを追加

## DDD / クリーンアーキテクチャ観点のメモ

### ドメインモデルへの影響

| 提案 | レイヤー配置 | ドメインモデルへの影響 |
|------|------------|---------------------|
| ブラウザ拡張 | プレゼンテーション層（新規パッケージ） | なし。既存APIを利用するため、ドメイン層の変更は不要 |
| AI自動タグ付け | ドメインサービス + インフラ層 | `TagSuggester` インターフェースをドメイン層に追加。`clipArticle` ユースケースを拡張 |
| ハイライト | ドメイン層（新エンティティ） | `Highlight` エンティティの新設。`Article` 集約との関係性を慎重に設計（別集約が望ましい） |
| 全文検索 | インフラ層 + アプリケーション層 | `ArticleContent` 値オブジェクトをドメイン層に追加。クエリサービスの拡張 |
| RSS購読 | 全レイヤー（新集約） | `Feed` 集約を新設。Cron実行のためのアプリケーションサービスを追加 |
| AIチャット | インフラ層 + アプリケーション層 | `ArticleAssistant` ドメインサービスインターフェースを追加 |
| エクスポート | アプリケーション層 + インフラ層 | `ExportService` をアプリケーション層に追加。ドメイン層への影響は軽微 |
| マルチメディア | ドメイン層（既存拡張） | `Source` 値オブジェクトにyoutube/pdfを追加。`ContentExtractor` インターフェースの拡張 |

### 設計上の注意点

- **ハイライト機能**: `Article` 集約にハイライトを含めると集約が肥大化するリスクがある。ハイライトは `Highlight` として独立した集約にし、`ArticleId` をID参照で持たせる設計が望ましい（集約間は結果整合性）
- **RSS購読**: `Feed` は `Article` とは独立した集約。フィードのエントリ取得 → `clipArticle` コマンド発行の流れで、既存のドメインモデルを再利用する
- **AI機能の拡張**: 現在の `ArticleSummarizer` と同様に、新しいAI機能もドメインサービスインターフェース（ドメイン層）+ Workers AI実装（インフラ層）のパターンを踏襲する
