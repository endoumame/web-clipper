# web-clipper 新機能提案 - 2026-04-01

## エグゼクティブサマリー

2026 年の Web クリッパー市場では AI 統合・ブラウザ拡張・ハイライト機能が標準化しつつあります。本プロジェクトは Clean Architecture/DDD に基づく堅牢なバックエンドを持ちますが、ブラウザ拡張の不在と AI 機能の限定性が競合との最大のギャップです。本提案では、ブラウザ拡張の開発、AI 自動タグ付け、ハイライト・アノテーション機能を軸に 8 つの機能を提案します。

## 現状の機能セット

- 記事クリッピング: URL からメタデータ（OGP、タイトル、説明文）を自動抽出し保存する
- AI 要約生成: Cloudflare AI（llama-3.1-8b-instruct）による記事要約
- タグ管理: カスタムタグによる記事分類（多対多リレーション）
- 検索・フィルタ: タイトル/説明文の全文検索やソース/タグ/既読状態フィルタに対応
- ソース自動分類: Twitter や Qiita、Zenn、GitHub など 11 種のソースに対応
- 認証: ユーザー名/パスワード + GitHub OAuth、Cookie ベースセッション管理
- PWA 対応: Web Share Target API による OS 共有メニューからの記事追加
- ダーク/ライトモード: テーマ切り替え・永続化
- View Transitions: ページ遷移アニメーション
- API ドキュメント: OpenAPI 仕様 + Scalar UI による API リファレンス

### 2026 年のトレンド

1. AI アシスタント統合: 単純な要約を超え、チャット型のドキュメント質問応答やコンテキスト理解に進化している
2. ハイライト・アノテーション: テキスト選択 → ハイライト保存が標準機能化した。再訪問時の可視化も重要である
3. ブラウザ拡張の必須化: ワンクリック保存が UX の基本である。全主要競合が Chrome/Firefox/Safari 拡張を提供している
4. プライバシー・ローカルファースト: Obsidian のようなローカル保存・オープンフォーマット志向が広がっている
5. 自動整理・スマートタグ: AI による自動カテゴリ分類・タグ付けでユーザーの手間を削減できる
6. マルチメディア対応: YouTube 動画や PDF、EPUB、ニュースレターなど URL 以外のソース対応が進んでいる

## 新機能提案

### 優先度: High（すぐに着手すべき）

#### 提案 1: ブラウザ拡張（Chrome Extension）

- 概要: Chrome 拡張機能を開発し、閲覧中のページをワンクリックでクリッピングする。選択テキストの右クリックメニューからの保存にも対応する
- ユーザー価値: 現在は Web UI から手動で URL を入力する必要がある。拡張により閲覧フローを中断せずに記事を保存できる。全主要競合が提供している機能であり、UX の根幹を成す
- 実装難易度: Medium
- 技術的アプローチ:
  - TypeScript で Chrome Extension Manifest V3 を採用する
  - `packages/extension` として monorepo に追加する
  - 既存の `POST /api/articles` エンドポイントをそのまま利用する
  - `@web-clipper/shared` の Zod スキーマを拡張側でも再利用し型安全性を確保する
  - Popup UI は Vue 3 で構築する（既存スタックの統一）
  - Content Script で選択テキスト・ページメタデータを取得する
  - 認証は Cookie ベースのセッションを拡張からも利用する（same-origin）。または API トークン方式を追加する
- 参考: [Obsidian Web Clipper (GitHub)](https://github.com/obsidianmd/obsidian-clipper), [Summer Web Clipper](https://github.com/merrickluo/summer-web-clipper)

#### 提案 2: AI 自動タグ付け・スマート分類

- 概要: 記事クリッピング時に AI が記事内容を解析し、既存タグから最適なタグを自動提案する。または新規タグを推薦する機能である
- ユーザー価値: 手動タグ付けの手間を削減する。記事が増えるほどタグ付けが面倒になる問題を解決し、整理された状態を維持できる。競合（Remio, Stacks, Notix）が実装済みの機能である
- 実装難易度: Low
- 技術的アプローチ:
  - 既存の Cloudflare AI binding（`env.AI`）を活用する
  - `clipArticle` コマンド実行時に、抽出済みコンテンツと既存タグリストをプロンプトに渡す
  - ドメイン層に `TagSuggester` ドメインサービスインターフェースを追加する
  - インフラ層に `WorkersAiTagSuggester` を実装する
  - API レスポンスに `suggestedTags: string[]` を追加し、フロントエンドで確認 UI を表示する
  - `@web-clipper/shared` にスキーマを追加する
- 参考: [Remio AI Features](https://www.remio.ai/post/top-web-clipper-tools-compared-find-your-perfect-match-in-2026), [Stacks AI-powered recall](https://chromewebstore.google.com/detail/stacks-web-clipper-and-ai/nfjbjlpkfimhobegkcoekpkdlokjkcfj)

### 優先度: Medium（次のマイルストーンで検討）

#### 提案 3: ハイライト・アノテーション機能

- 概要: 記事本文内の重要な箇所をハイライト（選択・色付け）し、メモを付与できる機能である。現在の `memo` フィールド（記事全体に 1 つ）を拡張し、本文中の特定箇所に紐づくアノテーションを実現する
- ユーザー価値: Readwise Reader や Obsidian Web Clipper、Web Highlights が全て提供する機能である。記事を「読んで終わり」ではなく「知識として蓄積」するための中核機能である
- 実装難易度: High
- 技術的アプローチ:
  - D1 に `highlights` テーブルを追加する（article_id, text_content, start_offset, end_offset 等）
  - ドメイン層に `Highlight` エンティティを新設する。`Article` 集約と分離して別集約にする
  - フロントエンドで記事詳細ページにテキスト選択 → ハイライト保存 UI を実装する
  - 将来的にブラウザ拡張の Content Script と連携し、元ページ上でのハイライトも可能にする
- 参考: [Readwise Reader Highlights](https://readwise.io/read), [Web Highlights](https://web-highlights.com/blog/top-7-web-clipper-chrome-extensions-for-capturing-websites/)

#### 提案 4: 全文コンテンツ検索

- 概要: 現在の検索はタイトル・説明文のみが対象である。Readability で抽出した記事本文を保存し、全文検索を可能にする
- ユーザー価値:「あの記事に書いてあった用語」を見つけたいとき、タイトルだけでは不十分である。記事本文をフルテキスト検索できることで知識ベースとしての価値が飛躍的に向上する
- 実装難易度: Medium
- 技術的アプローチ:
  - D1 に `article_content` テーブルを追加する（article_id, markdown_content, plain_text）
  - `clipArticle` コマンド内で既に `ContentExtractor` が Markdown を生成しているので、それを DB に永続化する
  - SQLite の FTS5（Full-Text Search）を活用する。D1 が FTS5 未対応の場合は LIKE 検索やセマンティック検索を検討する
  - 検索クエリ API に `contentSearch` パラメータを追加する
- 参考: [Readwise Reader Full-text Search](https://readwise.io/read)

#### 提案 5: RSS フィード購読

- 概要: RSS フィード URL を登録し、新着記事を自動でクリッピングする機能である。Cron Trigger で定期実行する
- ユーザー価値: お気に入りのブログ・メディアの新着記事を手動保存する手間を省ける。Readwise Reader が統合済みの機能である
- 実装難易度: Medium
- 技術的アプローチ:
  - D1 に `feeds` テーブル（id, url, title, last_fetched_at）と `feed_entries` テーブルを追加する
  - Cloudflare Workers の Cron Trigger（`wrangler.json` の `triggers.crons`）で定期実行する
  - 既存の `clipArticle` コマンドを再利用してフィードエントリをクリッピングする
  - ドメイン層に `Feed` 集約を新設する
- 参考: [Readwise Reader RSS](https://docs.readwise.io/reader/docs)

### 優先度: Low（長期的に検討）

#### 提案 6: AI チャットアシスタント（Ghostreader 的機能）

- Readwise Reader の Ghostreader のように、保存した記事に対して質問応答できるチャット機能である
- Cloudflare AI の LLM を活用し、記事コンテンツをコンテキストとして渡す
- 「この記事の主要な論点」「この技術の利点と欠点」などについて回答する

#### 提案 7: エクスポート機能（Obsidian / Notion 連携）

- 保存記事を Markdown 形式でエクスポートする（Obsidian Vault 連携）
- Notion API を通じた Notion データベースへの同期を実現する
- 一括エクスポート（JSON / CSV / Markdown ZIP）に対応する

#### 提案 8: マルチメディア対応（YouTube・PDF）

- YouTube URL からの動画メタデータ + 自動文字起こし保存に対応する
- PDF ファイルのアップロード・テキスト抽出・保存に対応する
- `Source` 値オブジェクトに youtube, pdf を追加する

## DDD / クリーンアーキテクチャ観点のメモ

### ドメインモデルへの影響

| 提案 | レイヤー配置 | ドメインモデルへの影響 |
|------|------------|---------------------|
| ブラウザ拡張 | プレゼンテーション層（新規パッケージ） | なし。既存 API を利用するため、ドメイン層の変更は不要 |
| AI 自動タグ付け | ドメインサービス + インフラ層 | `TagSuggester` インターフェースをドメイン層に追加。`clipArticle` ユースケースを拡張 |
| ハイライト | ドメイン層（新エンティティ） | `Highlight` エンティティの新設。`Article` 集約との関係性を慎重に設計（別集約が望ましい） |
| 全文検索 | インフラ層 + アプリケーション層 | `ArticleContent` 値オブジェクトをドメイン層に追加。クエリサービスの拡張 |
| RSS 購読 | 全レイヤー（新集約） | `Feed` 集約を新設。Cron 実行のためのアプリケーションサービスを追加 |
| AI チャット | インフラ層 + アプリケーション層 | `ArticleAssistant` ドメインサービスインターフェースを追加 |
| エクスポート | アプリケーション層 + インフラ層 | `ExportService` をアプリケーション層に追加。ドメイン層への影響は軽微 |
| マルチメディア | ドメイン層（既存拡張） | `Source` 値オブジェクトに youtube/pdf を追加。`ContentExtractor` インターフェースの拡張 |

### 設計上の注意点

- ハイライト機能: `Article` 集約にハイライトを含めると集約の肥大化リスクがある。ハイライトは `Highlight` として独立した集約にし、`ArticleId` を ID 参照で持たせる設計が望ましい（集約間は結果整合性）
- RSS 購読: `Feed` は `Article` とは独立した集約である。フィードのエントリ取得 → `clipArticle` コマンド発行の流れで、既存のドメインモデルを再利用する
- AI 機能の拡張: 現在の `ArticleSummarizer` と同様に、新しい AI 機能もドメインサービスインターフェースと Workers AI 実装のパターンを踏襲する
