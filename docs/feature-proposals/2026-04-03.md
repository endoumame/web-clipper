# web-clipper 新機能提案 - 2026-04-03

## エグゼクティブサマリー

前回提案（4/1）以降、AI 自動タグ付けが実装完了し、UI のモダナイズも進行中です。2026 年の競合調査から、対話型 AI アシスタントとブラウザ拡張が最も競争力に直結する機能だと判明しました。本提案では Readwise Ghostreader に対抗する AI チャット機能と、全競合が提供済みのブラウザ拡張を High 優先度として計 8 機能を提案します。

## 現状の機能セット

- 記事クリッピング — URL からメタデータ（OGP、タイトル、説明文）を自動抽出し保存する
- AI 要約生成 — Cloudflare AI（llama-3.1-8b-instruct）による記事要約
- AI 自動タグ付け — 4/1 以降に実装完了。クリッピング時に AI が既存タグから最適なタグを自動提案・適用する
- タグ管理 — カスタムタグの作成・記事への複数タグ付与、検索ボックス統合型タグフィルタ
- 検索・フィルタ — タイトル/説明文検索、ソース/タグ/既読フィルタ、カーソルベースページネーション
- ソース自動分類 — Twitter, Qiita, Zenn, GitHub 等 11 種のソースに対応
- 認証 — ユーザー名/パスワード + GitHub OAuth、Cookie ベースセッション
- PWA 対応 — Service Worker + Web Share Target API
- UI — ダーク/ライトモード切り替え、View Transitions、スクロール時ヘッダー非表示
- API ドキュメント — OpenAPI + Scalar UI

### 2026 年の重要トレンド

1. 対話型 AI の標準化 — 単発要約から「記事について質問できるチャット」へ進化している。Readwise Ghostreader や Mem が先行
2. マルチモデル対応 — Obsidian が OpenRouter 経由で任意プロバイダーを選択可能にした。ローカル LLM 対応はプライバシー訴求として有効
3. エージェント的自律行動 — MarkSnip の Agent Bridge や PixieBrix のワークフロー自動化に見られる「保存して終わり」からの脱却
4. YouTube/動画の第一級コンテンツ化 — 文字起こし・タイムスタンプ付きスクショ・要約を複数のツールがサポート
5. テンプレートエコシステム — Obsidian はドメインごとの自動適用テンプレートについてコミュニティリポジトリを運営
6. ローカルファースト — データをユーザーのデバイスから出さない設計が競争優位として明確化

## 新機能提案

### 🏆 優先度: High（すぐに着手すべき）

#### 提案 1: ブラウザ拡張（Chrome Extension Manifest V3）

閲覧中のページをワンクリックでクリッピングする Chrome 拡張を開発します。選択テキストの右クリック保存、ポップアップ UI でのタグ選択、AI 要約プレビューを含みます。

全主要競合が提供済みのため、ブラウザ拡張の不在は最大の UX ギャップです。Web UI への URL 手動入力を不要にし、クリッピングの摩擦をゼロに近づけます。

実装難易度は Medium です。

技術的アプローチは以下のとおりです。

- `packages/extension` として monorepo へ追加する。Manifest V3 + Vue 3 Popup 構成を採用
- Content Script で `document.title`、OGP メタタグ、選択テキストを取得する
- 既存の `POST /api/articles` を呼び出す。`@web-clipper/shared` の Zod スキーマを再利用する
- 認証は API トークン方式を新設する。拡張から Cookie 共有が困難なため `domain/api-token/` 集約を追加する
- Popup UI でタグ選択チップと AI 要約トグルを配置する
- Vite + CRXJS で開発・ビルド環境を構築する

参考は Obsidian Web Clipper（GitHub: obsidianmd/obsidian-clipper）と MarkSnip の Agent Bridge アーキテクチャです。

#### 提案 2: AI チャットアシスタント（記事への質問応答）

記事詳細ページにチャット UI を設け、保存した記事の内容へ自然言語で質問・対話できる機能を追加します。Readwise の Ghostreader に相当します。

長文記事の「特定の論点だけ知りたい」「要点を箇条書きにしたい」「翻訳してほしい」等のニーズへ対応します。記事を知識として活用するための中核機能です。

実装難易度は Medium です。

技術的アプローチは以下のとおりです。

- ドメイン層で `domain/article/article-assistant.ts` として `ArticleAssistant` インターフェースを追加する
- インフラ層で `WorkersAiArticleAssistant` を実装する。記事の Markdown コンテンツをシステムプロンプトとして含め、ユーザーの質問へ回答する
- 前提として提案 3（全文コンテンツ保存）の実装が必要
- アプリケーション層で `commands/ask-article-question.ts` コマンドを新設する
- プレゼンテーション層で `POST /api/articles/:id/chat` エンドポイントを追加する。SSE で逐次出力する
- フロントエンドでは `ArticleDetailPage` へチャットパネルを追加する。Vue の `ref` で会話履歴を管理する
- 会話履歴は初期段階ではクライアント側のみ保持する（DB 永続化は将来対応）

参考は Readwise Ghostreader チャット UI と Mem の拡張内 AI チャットです。

### 📋 優先度: Medium（次のマイルストーンで検討）

#### 提案 3: 記事本文の永続化と全文検索

`ContentExtractor` が生成する Markdown を DB へ保存し、記事本文を対象とした全文検索を実現します。AI チャット（提案 2）の前提機能でもあります。

「あの記事に書いてあった用語」をタイトルだけで探すのは困難です。本文検索により知識ベースとしての価値が飛躍的に向上します。

実装難易度は Medium です。

技術的アプローチは以下のとおりです。

- D1 で `article_contents` テーブルを追加する（article_id FK, markdown_content TEXT, plain_text TEXT）
- `clipArticle` コマンド内で `ContentExtractor` の出力を永続化する
- 検索は SQLite の LIKE + INSTR で初期実装し、D1 の FTS5 対応状況を確認のうえ利用可能なら移行する
- `ArticleQueryService` へ `contentSearch` パラメータを追加する
- ドメイン層で `ArticleContent` 値オブジェクトを追加する

参考は Readwise Reader の全文検索と Obsidian のローカル全文検索です。

#### 提案 4: ハイライト・アノテーション機能

記事詳細ページでテキストを選択してハイライト（色付け）し、選択箇所へメモを付与できる機能を追加します。

Web Highlights（20 万 DL, 4.83 星）や Readwise Reader が主力機能として提供しています。記事を「読んで終わり」から「知識として蓄積」へ変える中核機能です。

実装難易度は High です。

技術的アプローチは以下のとおりです。

- `Highlight` を独立した集約として設計する（`domain/highlight/`）。`ArticleId` を ID 参照で保持し `Article` 集約の肥大化を防ぐ
- D1 で `highlights` テーブルを追加する。カラムは id、article_id、text_content、color、note、start_offset、end_offset、created_at とする
- フロントエンドで `Selection API` + `Range API` を使いハイライト位置を特定・再現する
- `POST /api/articles/:id/highlights` エンドポイントを追加する

参考は Web Highlights と Readwise Reader ハイライト、Obsidian Web Clipper のページ内ハイライトです。

#### 提案 5: クリッピングテンプレートシステム

URL パターンや Schema.org データに基づき、サイトごとに異なるメタデータ抽出ルールを適用するテンプレート機能を実装します。Obsidian Web Clipper のテンプレートシステムに相当します。

Qiita なら「著者・いいね数・タグ」、GitHub なら「スター数・言語・README」など、サイト固有の情報を構造化して保存できます。手動入力の手間を減らし保存データの品質を向上させます。

実装難易度は Medium です。

技術的アプローチは以下のとおりです。

- ドメイン層で `domain/template/` へ `ClippingTemplate` 集約を新設する。`urlPattern`（正規表現）、`extractionRules`（JSON）、`name` を持つ
- インフラ層で `MetadataFetcher` を拡張する。テンプレートが存在する場合は CSS セレクタや Schema.org JSON-LD から追加データを抽出する
- D1 で `clipping_templates` テーブルを追加する
- Qiita、Zenn、GitHub、Twitter 等の既存 11 ソース向けにデフォルトテンプレートを事前定義する
- 将来的には AI による自然言語抽出ルール（Obsidian Interpreter 相当）を追加する

参考は Obsidian Web Clipper テンプレートと obsidian-community/web-clipper-templates です。

### 🔭 優先度: Low（長期的に検討）

#### 提案 6: マルチメディア対応（YouTube・PDF）

- YouTube URL からの動画メタデータと文字起こしを保存する。`Source` へ `youtube` を追加し、Cloudflare AI または外部 API で文字起こしを取得する
- PDF ファイルのアップロードとテキスト抽出をする。`pdf-parse` 等のライブラリを Workers 互換で利用する
- 動画や PDF を記事と同列で検索・タグ付け・AI 要約の対象とする

#### 提案 7: Markdown エクスポート・外部連携

- 保存記事を Markdown ZIP / JSON / CSV で一括エクスポートする
- Obsidian Vault 互換の Markdown 出力（YAML フロントマター付き）を提供する
- Notion API 経由のデータベース同期を将来対応する
- アプリケーション層で `ExportService` を追加する。ドメイン層への影響は軽微

#### 提案 8: RSS フィード購読と自動クリッピング

- RSS フィード URL を登録し、Cloudflare Workers Cron Trigger で定期取得する
- 新着エントリを自動で `clipArticle` コマンドへ流し、AI 要約と自動タグ付けも適用する
- `Feed` 集約を新設し、`Article` とは独立した集約として設計する

### 設計上の注意点

- AI チャットは全文保存に依存する。提案 2（AI チャット）は提案 3（全文保存）が前提のため、実装順序は 3 → 2 とする
- Highlight は独立集約とする。`Article` 集約へハイライトを含めると肥大化するため、`Highlight` は `ArticleId` を ID 参照で持つ独立集約として結果整合性で連携する
- API トークン認証を新設する。ブラウザ拡張向けの `ApiToken` 集約と既存 Cookie ベースセッションが共存する認証ミドルウェアを設計する
- テンプレートと Source の関係を整理する。現在の `Source` 列挙型（11 種）をテンプレートで吸収できれば、ソース追加時のドメイン層変更が不要になる
