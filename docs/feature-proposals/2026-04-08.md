# web-clipper 新機能提案 - 2026-04-08

## エグゼクティブサマリー

前回提案（4/1）の High 優先度 2 件（AI 自動タグ付け・ハイライト機能）が実装完了した。2026 年の Web クリッパー市場では AI が「要約」から「対話型 Q&A・セマンティック検索」へ進化している。本提案ではブラウザ拡張の実現と AI 機能の深化を軸に次のマイルストーンを定義する。

## 現状の機能セット

- 記事クリッピング: URL からメタデータ（OGP、タイトル、説明文）・本文を自動抽出し Markdown として保存
- AI 要約生成: Cloudflare Workers AI（llama-3.1-8b-instruct-fp8）による記事要約
- AI 自動タグ付け〔4/1 以降に実装〕: 記事クリッピング時、既存タグから最適なタグを自動提案・適用（`waitUntil` による非同期処理）
- ハイライト・アノテーション〔4/1 以降に実装〕: 記事本文のテキスト選択によるハイライト作成、カラー選択、メモ付与
- タグ管理: カスタムタグによる記事分類（多対多リレーション）、タグ別記事数表示
- 検索・フィルタ: タイトル/コンテンツの全文検索、ソース/タグ/既読状態フィルタ、カーソルベースページネーション
- ソース自動分類: Twitter、Qiita、Zenn、Hatena、GitHub、DevelopersIO、Medium、note、DEV、Stack Overflow、Other の 11 種
- 認証: ユーザー名/パスワード + GitHub OAuth、Cookie ベースセッション管理
- PWA 対応: Web Share Target API による OS 共有メニューからの記事追加
- ダーク/ライトモード: テーマ切り替え・永続化
- View Transitions: ページ遷移アニメーション
- API ドキュメント: OpenAPI 3.0.0 仕様 + Scalar UI による API リファレンス
- DDD / Clean Architecture: neverthrow による Result 型エラーハンドリング、ドメインモデルパターン

1. AI は要約から対話型 Q&A へと進化した。Readwise の Ghostreader がプロンプトボタン型から統合チャット UI へ移行している

## 新機能提案

### 🏆 優先度: High（すぐに着手すべき）

#### 提案 1: ブラウザ拡張（Chrome Extension）

Chrome 拡張機能を開発し、閲覧中のページをワンクリックでクリッピングする。選択テキストの右クリックメニューからのハイライト保存にも対応する。

全競合が提供する基本機能であり、Web UI からの手動 URL 入力は UX のボトルネックとなっている。拡張により閲覧フローを中断せずに記事保存・ハイライト作成が可能になる。

実装難易度は Medium です。

技術的なアプローチは以下のとおりです。

- `packages/extension` として monorepo に追加（Manifest V3）
- Popup UI は Vue 3 + UnoCSS で構築し、既存スタックを統一
- Content Script でページメタデータ・選択テキストを取得
- 既存の `POST /api/articles` と `POST /api/articles/:id/highlights` エンドポイントを利用
- `@web-clipper/shared` の Zod スキーマを拡張側でも再利用し型安全性を確保
- 認証は API トークン方式を追加（拡張は異なるオリジンのため Cookie 共有が困難）
- Vite でビルドし、Chrome Web Store への公開パイプラインを構築

参考: [Obsidian Web Clipper (GitHub)](https://github.com/obsidianmd/obsidian-clipper), [ClipAI](https://chromewebstore.google.com/detail/clipai-ai-web-clipper/flidicgbaijaifmdgkknkcefljnkhegc)

#### 提案 2: AI チャットアシスタント（記事 Q&A）

保存した記事に対して自然言語で質問できる対話型 AI 機能です。Readwise Ghostreader のように、記事内容をコンテキストとして渡し、要点抽出・質問応答・フラッシュカード生成します。

記事を「保存して終わり」ではなく「対話的に理解を深める」体験を提供する。Cloudflare Workers AI を既に利用しているため、追加コスト最小で実現できる。

実装難易度は Medium です。

技術的なアプローチは以下のとおりです。

- ドメイン層に `ArticleAssistant` ドメインサービスインターフェースを追加
- インフラ層に `WorkersAiArticleAssistant` を実装（既存の `WorkersAiArticleSummarizer` パターンを踏襲）
- 記事本文（Markdown）をシステムプロンプトのコンテキストとして渡し、ユーザーの質問に回答
- `POST /api/articles/:id/chat` エンドポイントを新設（ストリーミングレスポンス対応）
- 記事詳細ページにチャット UI を追加（スライドパネル形式）
- 会話履歴は D1 の `chat_messages` テーブルで永続化（オプション）
- `@web-clipper/shared` にチャット関連スキーマを追加

参考: [Readwise Ghostreader](https://readwise.io/read), [Mem Chat](https://productivitystack.io/guides/mem-ai-guide/)

### 📋 優先度: Medium（次のマイルストーンで検討）

#### 提案 3: セマンティック検索

現在のキーワードベース検索に加え、自然言語による意味的な検索を実現する。「先週読んだ React のパフォーマンス最適化の記事」のような曖昧なクエリで記事を検索できる。

記事が増えるほどキーワード検索では目的の記事にたどり着けない。ClipAI Pro や Mem.ai が既に提供しており、知識ベースとしての利用価値が向上する。

実装難易度は High です。

技術的なアプローチは以下のとおりです。

- Cloudflare Workers AI の埋め込みモデル（`@cf/baai/bge-small-en-v1.5` 等）でベクトル生成
- Cloudflare Vectorize をベクトルストアとして利用
- 記事クリッピング時に本文のベクトル埋め込みを生成・保存（`waitUntil` で非同期処理）
- `GET /api/articles` のクエリパラメータ `semanticQuery` を追加
- ドメイン層に `ArticleEmbedder` ドメインサービスインターフェースを追加
- フロントエンドの検索 UI で「AI 検索」トグルを追加

参考: [ClipAI セマンティック検索](https://chromewebstore.google.com/detail/clipai-ai-web-clipper/flidicgbaijaifmdgkknkcefljnkhegc), [Firecrawl Semantic Search API](https://www.firecrawl.dev/blog/best-semantic-search-apis)

#### 提案 4: RSS フィード購読

RSS フィード URL を登録し、新着記事を自動でクリッピングする。Cloudflare Workers の Cron Trigger で定期実行する。

お気に入りのブログ・技術メディアの新着記事を手動保存する手間を省ける。Readwise Reader が統合済みの主要機能であり、記事の入口を拡大する。

実装難易度は Medium です。

技術的なアプローチは以下のとおりです。

- D1 に `feeds` テーブル（id, url, title, lastFetchedAt）と `feed_entries` テーブルを追加
- ドメイン層で `Feed` 集約を新設（`Article` 集約とは独立）
- Cloudflare Workers の Cron Trigger（`wrangler.json` の `triggers.crons`）で定期実行（例: 1 時間ごと）
- 既存の `clipArticle` コマンドを再利用してフィードエントリをクリッピング
- API エンドポイント: `GET /api/feeds`, `POST /api/feeds`, `DELETE /api/feeds/:id`
- フロントエンドでフィード管理ページを追加

参考: [Readwise Reader RSS](https://docs.readwise.io/reader/docs)

#### 提案 5: エクスポート機能（Obsidian / Notion 連携）

保存記事・ハイライトを Markdown 形式でエクスポートし、Obsidian Vault や Notion データベースとの連携を実現する。

ベンダーロックインを防ぎ、既存のナレッジ管理ワークフローとの統合が可能になる。ローカルファースト志向のユーザーへ訴求する。

実装難易度は Low です。

技術的なアプローチは以下のとおりです。

- アプリケーション層で `ExportService` を追加（ドメイン層への影響は軽微）
- 単一記事エクスポート: `GET /api/articles/:id/export?format=markdown`
- 一括エクスポート: `GET /api/articles/export?format=zip`（JSON / CSV / Markdown ZIP）
- ハイライト・メモ・タグを含めた Markdown テンプレートを生成
- フロントエンドで「エクスポート」ボタンを追加

### 🔭 優先度: Low（長期的に検討）

#### 提案 7: API トークン認証

- D1 で `api_tokens` テーブルを追加（userId, token, name, expiresAt）

#### 提案 8: Daily Review / スペースドリピティション

- Readwise の Daily Review のように、過去のハイライトを定期的に表示して記憶定着を促進

- 将来的に TTS で音声化（Readwise のポッドキャスト的体験）

## DDD / クリーンアーキテクチャ観点のメモ

### ドメインモデルへの影響

| 提案 | レイヤー配置 | ドメインモデルへの影響 |
|------|------------|---------------------|
| ブラウザ拡張 | プレゼンテーション層（新規パッケージ） | なし。既存 API を利用するため、ドメイン層の変更不要。API トークン認証のみインフラ層へ追加 |
| AI チャット | ドメインサービス + インフラ層 | `ArticleAssistant` インターフェースをドメイン層へ追加。`ArticleSummarizer` と同様のパターン |
| セマンティック検索 | インフラ層 + クエリサービス | `ArticleEmbedder` インターフェースを追加。`ArticleQueryService` を拡張 |
| RSS 購読 | 全レイヤー（新集約） | `Feed` 集約を新設。`FeedRepository` を追加。`clipArticle` コマンドは再利用 |
| エクスポート | アプリケーション層 + インフラ層 | `ExportService` をアプリケーション層で追加。ドメイン層への影響は軽微 |
| マルチメディア | ドメイン層（既存拡張） | `Source` 値オブジェクトへ youtube/pdf を追加。`ContentExtractor` インターフェースの拡張 |
| API トークン | インフラ層 + プレゼンテーション層 | `ApiToken` エンティティを新設。認証ミドルウェアの拡張 |
| Daily Review | アプリケーション層 + インフラ層 | `ReviewService` をアプリケーション層で追加。Cron 実行基盤の構築 |

### 設計上の注意点

AI チャットでは `ArticleAssistant` を `ArticleSummarizer` と共通のインフラ基盤（Workers AI binding）上で実装する。DI コンテナ（`createDeps`）で注入するパターンを踏襲すること。

セマンティック検索では、ベクトル埋め込みの生成を `clipArticle` コマンド内の `waitUntil` で非同期実行する。既存の `suggestAndApplyTags` パターンを踏襲する。

RSS 購読では `Feed` を `Article` とは独立した集約にする。フィードエントリ取得後に `clipArticle` コマンドを発行し、既存のドメインモデルを再利用する。Cron Trigger はプレゼンテーション層の scheduled handler として実装する。

API トークンは提案 1（ブラウザ拡張）の前提機能です。認証ミドルウェアを拡張し、セッション認証とトークン認証を OR 条件で判定します。
