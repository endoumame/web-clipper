# web-clipper 新機能提案 - 2026-04-05

## エグゼクティブサマリー

前回提案（2026-04-01）以降に AI 自動タグ付け機能の実装が完了した。コア AI 機能の強化を踏まえ競合ツールとのギャップ分析を更新する。
次のステップとしてハイライト・アノテーション機能と Chrome 拡張機能を最優先に計 8 件を提案する。

## 現状の機能セット

- 記事クリッピング -- URL からメタデータ（OGP、タイトル、説明文）を自動抽出し保存
- AI 要約生成 -- Cloudflare Workers AI（llama-3.1-8b-instruct）による記事要約
- AI 自動タグ付け(新規実装済み) -- 記事保存時にバックグラウンドで AI がタグを自動提案・適用
- タグ管理 -- カスタムタグによる記事分類（多対多リレーション）
- 検索・フィルタ -- タイトル/説明文の全文検索とソース/タグ/既読状態フィルタ
- ソース自動分類 -- Twitter/X や Qiita、Zenn、Hatena、GitHub など 11 種に対応
- 認証 -- ユーザー名/パスワードと GitHub OAuth による Cookie ベースセッション管理
- PWA 対応 -- Web Share Target API による OS 共有メニューからの記事追加
- ダーク/ライトモード -- テーマ切り替えと永続化
- View Transitions -- ページ遷移アニメーション
- API ドキュメント -- OpenAPI 仕様と Scalar UI による API リファレンス

### 市場トレンド

1. AI チャット型アシスタント -- Readwise の Ghostreader に代表される対話型 AI が標準化しつつある
2. ハイライト・アノテーション -- Web Highlights（20 万 DL）が示すようにテキストハイライトの需要は高い
3. マルチフォーマット対応 -- PDF や YouTube 動画（文字起こし）への対応が差別化要因になっている
4. セマンティック検索 -- キーワードではなく意味に基づく検索が有料プランの中核機能になっている
5. ブラウザ拡張 -- Obsidian Clipper（30 万 DL）が示すようにブラウザからの直接保存が採用の鍵を握る
6. エクスポート・連携 -- Webhook や JSON/Markdown エクスポートなど外部ツール連携が充実してきている

## 新機能提案

### 🏆 優先度: High（すぐに着手すべき）

#### 提案 1: ハイライト・アノテーション機能

- 概要 -- 保存した記事内のテキストをハイライトしメモを付与できる機能
- ユーザー価値 -- 重要な箇所だけを後から参照でき学習・リサーチ効率を向上させる。Web Highlights（20 万 DL, 4.83 星）の実績が需要を裏付ける
- 実装難易度 -- Medium
- 技術的アプローチ
  - DB に `highlights` テーブルを追加する。カラムは article_id / start_offset / end_offset / text / note / color
  - ドメイン層に `Highlight` エンティティと `HighlightId` 値オブジェクトを追加する
  - `HighlightRepository` インターフェースをドメイン層に定義する
  - API として `POST /api/articles/{id}/highlights` と `DELETE /api/highlights/{id}` を追加する
  - フロントエンドでは Selection API と CSS Custom Highlight API（Chrome 105+）を活用する
  - Zod スキーマを `packages/shared` に追加し型安全性を維持する
- 参考 -- [Web Highlights](https://web-highlights.com/) / [Readwise Reader](https://readwise.io/read)

#### 提案 2: Chrome ブラウザ拡張機能

- 概要 -- 閲覧中の Web ページをワンクリックで保存できる Chrome 拡張機能
- ユーザー価値 -- 閲覧フローを中断せずにクリッピングできる。Obsidian Web Clipper が 30 万 DL を達成しており需要は実証済み
- 実装難易度 -- Medium
- 技術的アプローチ
  - monorepo に `apps/extension/` を追加する（Manifest V3, TypeScript）
  - Popup UI は Vue 3 と UnoCSS で既存の Web アプリと統一する
  - 既存の `POST /api/articles` エンドポイントを利用する
  - Content Script でページのメタデータ（OGP, title）を取得しポップアップに渡す
  - ビルドは Vite と CRXJS（Chrome Extension 向け Vite プラグイン）を利用する
  - 将来的にハイライト機能と統合しページ上で直接ハイライトして保存も可能にする
- 参考 -- [Obsidian Web Clipper](https://obsidian.md/clipper) / [Notion Web Clipper](https://chromewebstore.google.com/detail/notion-web-clipper/knheggckgoiihginacbkhaalnibhilkk)

### 📋 優先度: Medium（次のマイルストーンで検討）

#### 提案 3: セマンティック検索（AI ベクトル検索）

- 概要 -- 記事のタイトルや本文をベクトル化し自然言語クエリで意味的に近い記事を検索する機能
- ユーザー価値 -- 曖昧なクエリでも目的の記事を見つけられる。Remio や Readwise Pro の有料機能として実績がある
- 実装難易度 -- Medium
- 技術的アプローチ
  - Cloudflare Vectorize と Workers AI Embeddings（bge-base-en-v1.5）を利用する
  - 記事保存時は `waitUntil` でバックグラウンドのベクトル化を実行する（自動タグ付けと同じパターン）
  - ドメイン層に `ArticleEmbeddingService` インターフェースを追加する
  - API は `GET /api/articles?semantic=true&q=...` として既存エンドポイントを拡張する
- 参考 -- [Remio AI スマート検索](https://www.remio.ai/post/top-web-clipper-tools-compared-find-your-perfect-match-in-2026)

#### 提案 4: エクスポート機能（Markdown / JSON / CSV）

- 概要 -- 保存した記事データを Markdown や JSON、CSV 形式でエクスポートする機能
- ユーザー価値 -- Obsidian や Notion へのデータ移行とバックアップに対応しベンダーロックインを回避する
- 実装難易度 -- Low
- 技術的アプローチ
  - API に `GET /api/articles/export?format=markdown|json|csv` エンドポイントを追加する
  - ドメイン層に `ExportService` インターフェースを追加する
  - フロントエンドでは Blob と download 属性でクライアントサイドダウンロードする
  - Markdown エクスポートは既存の Turndown 基盤を再利用する
- 参考 -- [ClipAI エクスポート](https://chromewebstore.google.com/detail/clipai-ai-web-clipper/flidicgbaijaifmdgkknkcefljnkhegc)

#### 提案 5: RSS フィード購読

- 概要 -- RSS/Atom フィード URL を登録し新着記事を自動クリッピングする機能
- ユーザー価値 -- ブログや技術サイトの更新を自動収集しクリッピングの手間を削減する
- 実装難易度 -- Medium
- 技術的アプローチ
  - DB に `feeds` テーブルを追加する（id, url, title, lastCheckedAt）
  - Cloudflare Workers Cron Triggers で定期的にフィードをフェッチする
  - 新着記事は既存の `clipArticle` コマンドを再利用して自動保存する
  - ドメイン層に `Feed` エンティティと `FeedRepository` を追加する
  - API として `POST /api/feeds` と `GET /api/feeds`、`DELETE /api/feeds/{id}` を追加する
- 参考 -- [Readwise Reader RSS 機能](https://readwise.io/read)

## DDD / クリーンアーキテクチャ観点のメモ

### ハイライト機能（提案 1）のレイヤー配置

`Highlight` エンティティは `Article` 集約のローカルエンティティとして配置する。
`Article` を通じてのみアクセスされ `Article` の削除時にカスケード削除される。
ただしハイライト単体の CRUD 頻度が高い場合は独立集約として切り出すことも検討する。

アプリケーション層には `addHighlight` と `removeHighlight` コマンドを追加する。
`Article` 集約経由でハイライトを操作する。
インフラ層には `D1HighlightRepository` を追加し既存の Drizzle ORM パターンに従う。

### Chrome 拡張（提案 2）のアーキテクチャ

拡張機能は既存 API のクライアントとして位置付ける。
ドメイン層・アプリケーション層には変更不要。
認証方式として API トークン（Personal Access Token）の追加を検討する。
拡張機能からの Cookie ベース認証はクロスオリジン制約のため代替手段を用意する。
`packages/shared` の Zod スキーマを拡張機能でも共有し型安全性を維持する。

### セマンティック検索（提案 3）の配置

ドメイン層に `ArticleEmbeddingService` インターフェースを追加する。
embedding の生成はドメインサービスの責務とする。
インフラ層には `CloudflareVectorizeEmbeddingService` として実装する。
クエリ層の `ArticleQueryService` にセマンティック検索メソッドを追加し CQRS パターンに従う。
