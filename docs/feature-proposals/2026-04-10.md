# web-clipper 新機能提案 - 2026-04-10

## エグゼクティブサマリー

前回提案（2026-04-01）以降、AI 自動タグ付けとハイライト・アノテーション機能の実装が完了しました。しかし競合（Readwise Reader, Obsidian Web Clipper）との最大差分である「ブラウザ拡張の不在」は依然として未解決です。本提案では、ブラウザ拡張の開発を最優先とし、AI チャットアシスタントの導入を新たに High 優先度に昇格させます。

## 現状の機能セット

- 記事クリッピング — URL からメタデータ（OGP、タイトル、説明文）を自動抽出し、Readability + Turndown で Markdown 変換して保存
- AI 要約生成 — Cloudflare AI（llama-3.1-8b-instruct-fp8）による記事要約（バックグラウンド実行）
- AI 自動タグ付け 🆕 — クリッピング時に AI が記事内容を解析し、既存タグから最適なタグを自動提案・適用
- ハイライト・アノテーション 🆕 — 記事本文テキストの選択から 5 色ハイライト保存、メモ付与、CRUD 完備
- タグ管理 — カスタムタグによる記事分類（多対多リレーション）
- 検索・フィルタ — タイトルのテキスト検索やソース・タグ・既読状態フィルタに対応
- ソース自動分類 — Twitter や Qiita、Zenn、GitHub 等 11 種以上のソースに対応
- 認証 — ユーザー名/パスワードと GitHub OAuth による Cookie ベースセッション管理
- UI/UX — ダーク/ライトモード切替や View Transitions API によるアニメーション
- API ドキュメント — OpenAPI 3.0.0 仕様 + Scalar UI による対話的 API リファレンス
- アーキテクチャ — Clean Architecture（4 層）と DDD に基づく設計。neverthrow による型安全エラーハンドリング

## 新機能提案

### 🏆 優先度: High（すぐに着手すべき）

#### 提案 1: ブラウザ拡張（Chrome Extension）

Chrome 拡張機能（Manifest V3）を開発し、閲覧中のページをワンクリックでクリッピングする。選択テキストのコンテキストメニュー保存やタグ選択・メモ入力にも対応する。

現在は Web UI から手動で URL を入力する必要があり、閲覧フローが中断される。全主要競合（Obsidian 30 万 DL、Notion 40 万 DL）が提供する機能であり、本プロジェクト最大の競合ギャップです。

実装難易度は Medium。

技術的なアプローチを以下に示します。

- `packages/extension` として monorepo に追加する。TypeScript + Manifest V3 を採用
- Popup UI は Vue 3 + UnoCSS で構築し、既存フロントエンドと統一する
- `@web-clipper/shared` の Zod スキーマを拡張側で再利用し、型安全な API 通信を実現
- Content Script でページメタデータ・選択テキストを取得する
- 既存の `POST /api/articles` エンドポイントをそのまま利用する（ドメイン層の変更不要）
- 認証は API トークン方式を追加する。Cookie ベースでは cross-origin 制約がある
- Service Worker でオフラインキューを管理し、ネットワーク復帰時に一括送信する

参考: [Obsidian Web Clipper (GitHub)](https://github.com/obsidianmd/obsidian-clipper), [Building a Web Clipper Chrome Extension (Medium, 2026)](https://khushalthepane2000.medium.com/building-a-web-clipper-chrome-extension-to-supercharge-my-developer-workflow-79bfee39b3c9)

#### 提案 2: AI チャットアシスタント（Ghostreader 的機能）

保存した記事に対してチャット形式で質問応答できる AI アシスタントを実装する。記事コンテンツをコンテキストとして渡し、要点抽出や用語解説などを対話的に行う。

Readwise Reader の Ghostreader v2 が統一チャット UI を導入し、Web Highlights も AI Chat を追加した。2026 年の市場では「保存した記事と対話できる」機能が標準化しつつある。

実装難易度は Medium。

技術的なアプローチを以下に示します。

- ドメイン層に `ArticleChat` ドメインサービスインターフェースを追加する
- インフラ層に `WorkersAiArticleChat` を実装する。Cloudflare AI を使用
- システムプロンプトに記事本文（Markdown）を含め、コンテキスト付きで回答する
- `POST /api/articles/{id}/chat` エンドポイントを新設する
- フロントエンドでは記事詳細ページにチャットパネルを追加する
- 将来的にストリーミングレスポンス（SSE）対応で UX を向上させる
- `@web-clipper/shared` に `ChatInputSchema`, `ChatResponseSchema` を追加する

参考: [Readwise Ghostreader](https://readwise.io/read), [Web Highlights AI Chat](https://web-highlights.com/blog/best-summary-website-ai-tools/)

### 📋 優先度: Medium（次のマイルストーンで検討）

#### 提案 3: 全文コンテンツ検索

現在の検索はタイトルのみが対象です。Readability で抽出済みの記事本文（Markdown）を活用し、全文検索を可能にする。

「あの記事に書いてあったキーワード」をタイトルだけでは見つけられない。知識ベースとしての価値が向上する。

実装難易度は Medium。

技術的なアプローチを以下に示します。

- 記事本文は既に `articles.content` カラムに保存されている。検索クエリで本文も対象に含める
- D1 が SQLite FTS5 に対応している場合は仮想テーブルを活用する。未対応の場合は `LIKE` ベースで対応
- 将来的に Cloudflare AI の Embedding + Vectorize によるセマンティック検索へ拡張する
- `ArticleQueryParamsSchema` に `contentSearch` パラメータを追加する

参考: [Readwise Reader Full-text Search](https://readwise.io/read)

#### 提案 4: エクスポート・データポータビリティ

保存記事・ハイライト・タグを複数フォーマット（Markdown, JSON, CSV）でエクスポートします。ハイライト付き Markdown は Obsidian Vault へ配置可能な形式で出力します。

データポータビリティは 2026 年のトレンドです。Linkflare が HTML/CSV/JSON エクスポートを提供し、Wallabag が EPUB/PDF/JSON に対応している。ベンダーロックインへの不安を解消できる。

実装難易度は Low。

技術的なアプローチを以下に示します。

- アプリケーション層に `ExportArticlesQuery` を追加する
- `GET /api/articles/export?format=markdown&tagId=xxx` エンドポイントを新設する
- Markdown エクスポートでは YAML frontmatter と本文、ハイライト注釈を含める
- JSON エクスポートでは全記事メタデータとハイライト、タグをすべて出力する
- フロントエンドの設定ページにエクスポートボタンを追加する
- ドメイン層への影響は軽微である（リードモデルの活用で十分）

参考: [Linkflare Data Export](https://linkflare.io/articles/read-later-app/), [Wallabag Export](https://www.readless.app/blog/best-read-later-apps-comparison)

#### 提案 5: RSS フィード購読

RSS フィード URL を登録し、新着記事を自動でクリッピングする。Cloudflare Workers の Cron Trigger で定期実行する。

お気に入りブログ・メディアの新着記事を手動保存する手間を削減できる。Readwise Reader が RSS リーダーを内蔵しており、read-it-later アプリの標準機能になりつつある。

実装難易度は Medium。

技術的なアプローチを以下に示します。

- D1 に `feeds` テーブルと `feed_entries` テーブルを追加する
- ドメイン層に `Feed` 集約を新設する（`FeedId`, `FeedUrl` 値オブジェクト）
- Cloudflare Workers Cron Trigger で定期実行する
- 既存の `clipArticle` コマンドを再利用してフィードエントリをクリッピングする
- フロントエンドにフィード管理ページを追加する

参考: [Readwise Reader RSS](https://docs.readwise.io/reader/docs)

### 🔭 優先度: Low（長期的に検討）

#### 提案 6: マルチメディア対応（YouTube・PDF）

- YouTube URL からの動画メタデータと自動文字起こし（Cloudflare AI Whisper）保存に対応する
- PDF ファイルのアップロードとテキスト抽出（`pdf.js`）に対応する

#### 提案 7: セマンティック検索（AI ベクトル検索）

- Cloudflare AI の Embedding モデル + Vectorize（ベクトル DB）を活用した意味検索
- 「機械学習の入門記事」のような自然言語クエリで関連記事を検索可能にする
- 全文検索（提案 3）の上位互換として段階的に導入する

#### 提案 8: TTS（テキスト読み上げ）対応

- Readwise Reader が Daily Review の TTS 対応を導入した。保存記事を音声で「聴く」体験を提供する
- Cloudflare AI の TTS モデルまたは Web Speech API を活用する
- 通勤中や運動中の記事消化を可能にし、利用シーンを拡大する

## DDD / クリーンアーキテクチャ観点のメモ

### ドメインモデルへの影響

| 提案 | レイヤー配置 | ドメインモデルへの影響 |
|------|------------|---------------------|
| ブラウザ拡張 | プレゼンテーション層（新規パッケージ） | ドメイン層の変更不要。API トークン認証を追加する場合は `PersonalAccessToken` を新設 |
| AI チャット | ドメインサービス + インフラ層 | `ArticleChat` ドメインサービスインターフェースを追加 |
| 全文検索 | インフラ層 + アプリケーション層 | ドメイン層への影響なし。クエリサービスの拡張のみ |
| エクスポート | アプリケーション層 | ドメイン層への影響は軽微。リードモデルからの組み立てで十分 |
| RSS 購読 | 全レイヤー（新集約） | `Feed` 集約を新設。`Article` との間は結果整合性 |
| マルチメディア | ドメイン層（既存拡張） | `Source` 値オブジェクトと `ContentExtractor` の拡張 |
| セマンティック検索 | インフラ層 | ドメイン層への影響なし。ベクトル DB アダプタを追加 |
| TTS | インフラ層 + アプリケーション層 | `ArticleReader` ドメインサービスインターフェースを追加 |

### 設計上の注意点

ブラウザ拡張の認証について — Cookie ベースセッションは cross-origin 制約がある。Personal Access Token（PAT）方式を追加し、`User` 集約の外（別集約）として `UserId` で ID 参照する設計が望ましい。

AI チャットのコンテキスト管理について — 記事本文が長い場合は LLM のトークン制限に注意が必要です。チャンク分割やサマリーチェーンの設計をインフラ層に閉じ込める。

RSS フィードと Article の関係について — ドメインイベントで疎結合にするのが理想的です。ただし Cloudflare Workers のイベント基盤は限定的なため、アプリケーションサービスで `clipArticle` を直接呼び出す方が現実的です。
