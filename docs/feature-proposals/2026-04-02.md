# web-clipper 新機能提案 - 2026-04-02

## エグゼクティブサマリー

先週の AI 自動タグ付け実装を踏まえ、今週はコンテンツの「蓄積」から「活用」へのシフトを提案します。
全文検索・ハイライト・AI チャットアシスタントを軸に、保存した記事を知識ベースとして活用できる機能群を提案します。
加えて、2026 年のトレンドに沿ったコンテキスト付き保存とスマートリコール機能を新たに提案します。

## 現状の機能セット

- 記事クリッピング — URL からメタデータ（OGP、タイトル、説明文）を自動抽出し保存
- AI 要約生成 — Cloudflare AI（llama-3.1-8b-instruct）による記事要約
- AI 自動タグ付け〔新規実装済み〕 — 記事クリッピング時に AI がタグを自動で提案・適用（`TagSuggester` ドメインサービス）
- タグ管理 — カスタムタグによる記事分類（多対多リレーション）。検索ボックス統合型タグフィルターも追加済み
- 検索・フィルタ — タイトルおよび説明文のテキスト検索、ソース/タグ/既読状態フィルタに対応
- ソース自動分類 — Twitter / Qiita / Zenn / GitHub など 11 種のソースに対応
- 認証 — ユーザー名/パスワード + GitHub OAuth、Cookie ベースのセッション管理に対応
- PWA 対応 — Web Share Target API による OS 共有メニューからの記事追加に対応
- ダーク/ライトモード — テーマ切り替えと永続化に対応
- View Transitions — ページ遷移アニメーション
- モダン UI〔新規〕 — ヘッダー・検索入力・ソースフィルターの UI を刷新済み
- API ドキュメント — OpenAPI 仕様 + Scalar UI による API リファレンス

## 新機能提案

### 🏆 優先度 High（すぐに着手すべき）

#### 提案 1: ブラウザ拡張（Chrome Extension）

Chrome 拡張機能を開発し、閲覧中のページをワンクリックで保存します。
選択テキストの右クリックメニュー保存やポップアップからのタグ選択にも対応します。

現状は Web UI から URL を手動入力しなければなりません。
拡張により閲覧フローを中断せずに記事を保存できます。
全主要競合が提供済みの基本機能です。

実装の難易度は Medium です。

技術的アプローチは以下の通りです。

- TypeScript + Chrome Extension Manifest V3 を採用
- `packages/extension` として monorepo に追加
- 既存の `POST /api/articles` エンドポイントをそのまま利用
- `@web-clipper/shared` の Zod スキーマを拡張側でも共有し型安全性を確保
- Popup UI は Vue 3 + 既存のデザインシステムで構築
- Content Script でページメタデータ・選択テキストを取得
- 実装済みの AI 自動タグ付け（`suggestAndApplyTags`）を拡張からも呼び出す

参考:
[Obsidian Web Clipper](https://obsidian.md/clipper),
[PixieBrix Context-Aware Clipper](https://blog.pixiebrix.com/blog/the-best-web-clippers-in-2026-capture-the-web-without-losing-context)

#### 提案 2: 記事の本文を永続化して全文検索を実現

`ContentExtractor` が抽出した Markdown の本文を D1 に永続化します。
記事の本文を含む全文検索を可能にします。

現在の検索はタイトル・説明文のみが対象です。
「あの記事に書いてあった用語」を見つけられません。
本文の検索により知識ベースとしての価値が飛躍的に向上します。

実装の難易度は Medium です。

技術的アプローチは以下の通りです。

- D1 に `article_contents` テーブルを追加（`article_id`, `markdown_content`, `plain_text`）
- `clipArticle` コマンド内で既に Markdown を生成しているため、それを永続化
- D1 の SQLite FTS5 の対応状況を確認し、使えない場合は `LIKE` 検索を検討
- 検索 API に `fulltext` パラメータを追加し、フロントエンドの検索 UI を拡張
- ドメイン層に `ArticleContent` 値オブジェクトを追加

参考:
[Readwise Reader Full-text Search](https://readwise.io/read),
[Remio AI Smart Search](https://www.remio.ai/post/top-web-clipper-tools-compared-find-your-perfect-match-in-2026)

### 📋 優先度 Medium（次のマイルストーンで検討）

#### 提案 3: ハイライトとアノテーションの機能

記事詳細ページでテキストを選択してハイライト・メモを保存する機能を追加します。
保存したハイライトの一覧表示・検索にも対応します。

記事を「読んで終わり」ではなく「知識として蓄積」するための中核的な機能です。
Readwise Reader や Web Highlights、Obsidian が全て提供済みです。

実装の難易度は High です。

技術的アプローチは以下の通りです。

- D1 に `highlights` テーブルを追加（`id`, `article_id`, `text_content`, `color` 等）
- ドメイン層に `Highlight` エンティティを新設し、`Article` 集約とは分離した独立の集約にする
- フロントエンドで記事のテキスト選択からハイライト保存の UI を実装（`Selection API` を利用）
- ハイライト一覧ページを追加し、記事を横断してハイライトを閲覧・検索できるようにする

参考:
[Readwise Reader Highlights](https://readwise.io/read),
[Web Highlights](https://web-highlights.com/blog/best-summary-website-ai-tools/)

#### 提案 4: AI チャットアシスタント（記事への質問応答）

保存した記事に対してチャット形式で質問できる AI アシスタントを提供します。
Readwise Reader の Ghostreader v2 に相当する機能です。

長い技術記事の要点を確認したり、専門用語の解説を得たりできます。
2026 年のトレンドに沿ったチャット型ドキュメント AI の機能です。

実装の難易度は Medium です。

技術的アプローチは以下の通りです。

- 提案 2（記事の本文の永続化）を前提とする
- Cloudflare AI の LLM を利用し、記事のコンテンツをコンテキストとして渡す
- ドメイン層に `ArticleAssistant` ドメインサービスのインターフェースを追加
- インフラ層に `WorkersAiArticleAssistant` を実装
- フロントエンドの記事詳細ページにチャット UI を追加
- 既存の `ArticleSummarizer` パターンを踏襲し、一貫したアーキテクチャを維持

参考:
[Readwise Ghostreader v2](https://www.speedreadinglounge.com/readwise-reader-review),
[Stacks AI-powered recall](https://chromewebstore.google.com/detail/stacks-web-clipper-and-ai/nfjbjlpkfimhobegkcoekpkdlokjkcfj)

#### 提案 5: RSS フィード購読と自動クリッピング

RSS フィード URL を登録し、新着記事を自動でクリッピングします。
Cloudflare Workers の Cron Trigger で定期的に実行します。

お気に入りのブログやメディアの新着記事を手動で保存する手間を省けます。
Readwise Reader が RSS リーダーを統合済みです。

実装の難易度は Medium です。

技術的アプローチは以下の通りです。

- D1 に `feeds` テーブルと `feed_entries` テーブルを追加
- `wrangler.json` に Cron Trigger を追加し、定期実行のハンドラを実装
- 既存の `clipArticle` コマンドを再利用してフィードエントリをクリッピング
- ドメイン層に `Feed` 集約を新設
- フロントエンドにフィード管理ページを追加

参考:
[Readwise Reader RSS](https://docs.readwise.io/reader/docs),
[Remio Auto Capture](https://www.remio.ai/post/top-web-clipper-tools-compared-find-your-perfect-match-in-2026)

## DDD / クリーンアーキテクチャ観点のメモ

### ドメインモデルへの影響

| 提案 | レイヤー | 影響 |
|------|---------|------|
| ブラウザ拡張 | プレゼンテーション層 | なし。既存 API を再利用 |
| 全文検索 | インフラ層 + アプリ層 | `ArticleContent` 値オブジェクトを追加 |
| ハイライト | ドメイン層 | `Highlight` を独立した集約として新設 |
| AI チャット | ドメインサービス + インフラ層 | `ArticleAssistant` IF を追加 |
| RSS 購読 | 全レイヤー | `Feed` 集約を新設 |
| リコール | アプリ層 + インフラ層 | `ReviewScheduler` ドメインサービスを追加 |
| エクスポート | アプリ層 + インフラ層 | `ExportService` をアプリ層に追加 |
| セマンティック検索 | インフラ層 | `ArticleEmbedding` 値オブジェクトを追加 |

### 設計上の注意点

ハイライト集約の独立性について — `Article` 集約にハイライトを含めると集約が肥大化します。
`Highlight` は `ArticleId` を ID 参照で持つ独立した集約にし、結果整合性で管理します。

全文検索のリードとライトの分離について — 本文の永続化と検索は分離して設計します。
将来の CQRS パターンへの移行を見据えた構造にします。

AI サービスの統一パターンについて — `ArticleSummarizer` → `TagSuggester` → `ArticleAssistant` と一貫したパターンを適用します。
「ドメインサービス IF + Workers AI 実装」の構造を踏襲します。

RSS フィードの `clipArticle` 再利用について — 既存コマンドを再利用します。
AI 要約と自動タグ付けなど全ての後処理がフィード記事にも自動で適用されます。
