# web-clipper 新機能提案 - 2026-04-11

## エグゼクティブサマリー

前回提案（2026-04-01）以降、AI 自動タグ付けとハイライト機能の実装が完了しました。本提案では競合の最新動向を踏まえ、セマンティック検索とドキュメントチャット AI を最優先に、全 8 件の新機能を提案します。

## 現状の機能セット

- 記事クリッピング: URL からメタデータ（OGP、タイトル、説明文）を自動で抽出し Markdown で保存する
- AI 要約生成: Cloudflare Workers AI（Llama 3.1 8B）による記事の自動要約
- AI 自動タグ付け: 記事のクリップ時に既存タグを考慮した AI タグ提案の自動適用（`waitUntil()` で非同期に実行）
- ハイライト・アノテーション: 記事内テキストのハイライト作成（カラー指定）やノート付与に対応する。前後文脈も保存する
- タグ管理: カスタムタグによる記事分類（多対多リレーション）
- 検索・フィルタ: タイトルや説明文の検索、ソース・タグ・既読状態によるフィルタに対応する
- ソース自動分類: Twitter や GitHub、Qiita、Zenn など 11 種のソースに自動分類する
- 認証: ユーザー名/パスワード + GitHub OAuth、Cookie ベースセッション管理
- PWA 対応: Web Share Target API による OS 共有メニューからの記事追加
- ダーク/ライトモード: テーマ切り替え・localStorage への永続化
- View Transitions API: ページ遷移アニメーション
- API ドキュメント: Hono + zod-openapi による OpenAPI 仕様 + Scalar UI

## 新機能提案

### 優先度: High（すぐに着手すべき）

#### 提案 1: セマンティック検索（Vectorize + AI Embeddings）

- 概要: 記事コンテンツをベクトル化し、自然言語で意味的に類似する記事を検索可能にする
- ユーザー価値: 保存する記事が増えるほど「あの記事はどこだっけ」という問題が深刻になる。意味的な検索により、ナレッジベースとしての実用性が大きく向上する
- 実装難易度: Medium
- 技術的アプローチ:
  - Cloudflare Vectorize を利用してベクトルインデックスを作成する
  - Cloudflare Workers AI の Embedding モデル（`@cf/baai/bge-base-en-v1.5` 等）で記事をベクトル化する
  - 記事の保存時に `waitUntil()` でバックグラウンドのベクトル化を実行する。自動タグ付けと同じパターンである
  - Domain 層に `ArticleSearchService` インターフェースを定義し、Infrastructure 層で Vectorize の実装を配置する
  - 既存のキーワード検索と並行してセマンティック検索の結果をマージする Query を追加する
- 参考: Readwise Reader のセマンティック検索、Cloudflare Vectorize のドキュメント

#### 提案 2: ドキュメントチャット AI（記事に対する Q&A）

- 概要: 保存した記事に対して自然言語で質問し AI が回答する、対話型のインターフェースを提供する
- ユーザー価値: 長い記事を全文読む代わりに、知りたいポイントを直接質問できる。Readwise の Ghostreader チャットが示すように、ユーザーの期待は「保存して要約」から「保存して対話」へ移行している
- 実装難易度: Medium
- 技術的アプローチ:
  - 既存の `ArticleSummarizer` を参考に `ArticleChatService` インターフェースを Domain 層に定義する
  - Cloudflare Workers AI の Chat Completion API を利用する。記事コンテンツをコンテキストとして渡す
  - `POST /api/articles/:id/chat` エンドポイントを追加する。ストリーミングレスポンスに対応する
  - フロントエンドでは `ArticleDetailPage` 内にサイドパネルとしてチャット UI を追加する
  - Application 層に `ChatWithArticleCommand` を追加する
- 参考: Readwise Ghostreader Chat（2025 年 12 月に刷新）、Obsidian Interpreter

### 優先度: Medium（次のマイルストーンで検討）

#### 提案 3: ブラウザ拡張（Chrome Extension）

- 概要: Chrome 拡張によるワンクリック記事クリッピングである。閲覧中のページから記事を直接保存できる
- ユーザー価値:「URL をコピーして Web アプリを開きペーストする」フローをボタン 1 つに短縮する。競合ツールの標準的な UX である
- 実装難易度: High
- 技術的アプローチ:
  - Chrome Manifest V3 ベースの拡張を `apps/extension/` として monorepo に追加する
  - Popup UI は Vue 3 + Vite で構築する。既存 Web アプリのコンポーネントを共有する
  - Content Script で Readability によるコンテンツを抽出する
  - `packages/shared` の Zod スキーマを拡張でも利用し型安全性を維持する
- 参考: Obsidian Web Clipper（Manifest V3 で 30 万 DL）、Web Highlights

#### 提案 4: Markdown エクスポート

- 概要: 保存した記事をメタデータ付き Markdown ファイルとしてエクスポートする。単体・一括・タグ別に対応する
- ユーザー価値: データポータビリティはユーザーの信頼の基盤である。Obsidian 等への移行を容易にしベンダーロックインへの不安を解消する
- 実装難易度: Low
- 技術的アプローチ:
  - `GET /api/articles/export` エンドポイントを追加する。クエリパラメータでフィルタ条件を指定する
  - 記事の `content`（既に Markdown）に YAML frontmatter を付与する
  - 一括の場合は ZIP 圧縮する。Workers での ZIP 生成には軽量な `fflate` を使用する
  - Application 層に `ExportArticlesQuery` を追加する
- 参考: Obsidian Web Clipper の Markdown 出力形式、MarkDownload

#### 提案 5: RSS フィード購読（Cron Trigger で自動クリップ）

- 概要: RSS/Atom フィードを登録し Cron Trigger で定期的に新着記事を自動クリッピングする
- ユーザー価値: 手動クリップ作業を削減し情報収集を自動化する。Readwise Reader の RSS 統合に対抗する基本機能である
- 実装難易度: Medium
- 技術的アプローチ:
  - D1 に `feeds` テーブルと `feed_entries` テーブルを追加する
  - Domain 層に `Feed` エンティティと `FeedSubscription` 集約を定義する
  - Cloudflare Workers の `scheduled()` ハンドラで Cron Trigger（例: 毎時）を設定する
  - 新着エントリを検出したら既存の `ClipArticleCommand` を再利用して記事として保存する
- 参考: Readwise Reader の RSS 統合、Miniflux（セルフホスト RSS リーダー）

### 優先度: Low（長期的に検討）

#### 提案 6: スペースドリピティション（ハイライト復習システム）

- 保存したハイライトを間隔反復アルゴリズム（SM-2 等）に基づき定期的に通知し復習する
- `highlights` テーブルに `nextReviewAt` や `easeFactor`、`interval` カラムを追加する
- フロントエンドに「今日の復習」ページを追加し、カードフリップ UI で復習する
- 参考: Readwise のハイライト復習や Anki のアルゴリズム

#### 提案 7: YouTube トランスクリプト対応

- YouTube URL のクリップ時に自動で文字起こしを取得し記事コンテンツとして保存する
- トランスクリプトに対してもハイライトや AI 要約、チャットを適用できるようにする
- `source` に YouTube を追加し、サムネイルやトランスクリプトの専用 UI を提供する
- 参考: Readwise Reader の YouTube トランスクリプトハイライト

#### 提案 8: マルチ AI プロバイダ対応

- Cloudflare Workers AI に加え OpenAI や Anthropic、Ollama（ローカル）を選択可能にする
- Domain 層の `ArticleSummarizer` や `TagSuggester` インターフェースは変更不要である。DDD の恩恵である
- Infrastructure 層に `OpenAiSummarizer` や `AnthropicSummarizer` 等の実装を追加する
- ユーザー設定で API キーとモデルを選択する UI を提供する
- 参考: Obsidian Interpreter のマルチモデル対応、Readwise の GPT-5/o3 対応

## DDD / クリーンアーキテクチャ観点のメモ

### ドメインモデルへの影響

| 提案 | レイヤー配置 | ドメインモデルへの影響 |
|------|------------|---------------------|
| セマンティック検索 | Domain 層 + Infrastructure 層 | `ArticleSearchService` インターフェースを追加する |
| ドキュメントチャット | Domain 層 + Infrastructure 層 | `ArticleChatService` インターフェースを追加する |
| ブラウザ拡張 | Presentation 層（新規パッケージ） | ドメイン層の変更は不要である |
| Markdown エクスポート | Application 層 + Presentation 層 | `ExportArticlesQuery` を追加する |
| RSS 購読 | 全レイヤー | `Feed` 集約を新設する |
| スペースドリピティション | Domain 層 + Application 層 | `Highlight` エンティティに復習スケジュールを追加する |
| YouTube 対応 | Infrastructure 層 | `ContentExtractor` インターフェースを拡張する |
| マルチ AI プロバイダ | Infrastructure 層のみ | Domain 層の変更は不要である。アダプタを追加するのみ |

### 設計上の注意点

- セマンティック検索: ベクトルインデックスの更新は `ClipArticleCommand` 内で明示的に呼び出す。ドメインイベントは将来の検討事項とする
- ドキュメントチャット: チャット履歴は短命なため集約としてモデリングする必要はない。Application 層で管理する
- RSS 購読: `Feed` は `Article` とは独立した集約である。フィードの新しいエントリを取得した後、`ClipArticleCommand` を呼び出して記事を保存する。集約どうしは直接参照しない
- マルチ AI プロバイダ: DDD のポートとアダプタパターンの恩恵を最も受ける提案である。Domain 層は不変のまま Infrastructure 層にアダプタを追加するだけで対応できる
